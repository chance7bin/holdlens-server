## 1. 背景

当前 agent 项目已经具备基金公开数据刷新、结构化快照和 Markdown 报告生成能力。下一阶段需要由 server 作为业务系统和长期事实源，通过后台任务主动发起基金信息刷新，把需要刷新的基金代码发送给 agent；agent 完成公开数据刷新后回调 server；server 保存全局基金公开数据结果，并通过接口提供账户资产和对应基金明细。

这意味着 server 不再把 agent 本地生成的 `finance/基金明细跟踪.md` 作为查询事实源，也不要求 agent 为 server 集成链路继续生成 Markdown 报告。Markdown 可以保留为本地工具输出，但不参与 server 的业务查询链路。

## 2. 目标

- server 可以通过后台调度或系统任务触发基金明细刷新任务。
- server 只向 agent 发送必要的基金代码和任务上下文。
- agent 刷新完成后回调 server，server 接收结构化基金详情结果。
- server 保存全局基金公开数据快照、刷新 warning 和任务状态。
- 调用方可以通过 server API 查询账户资产及其对应的最新基金明细。
- 刷新链路不依赖或生成 Markdown 报告作为业务输出。

## 3. 范围

### 本次包含

- server 通过后台调度或系统任务创建基金公开数据刷新任务。
- server 向 agent 下发基金代码列表、任务标识和回调地址。
- server 接收 agent 刷新结果回调，并展示任务成功、部分失败或失败状态。
- server 保存 agent 返回的结构化基金详情、数据来源和 warning。
- server 提供账户资产与基金明细组合查询能力。
- server 与 agent 之间保留必要的鉴权、幂等和错误语义要求。

### 本次不包含

- 不由 server 直接抓取外部基金行情或公开重仓数据。
- 不在 server 中实现东方财富、天天基金、AkShare provider。
- 不通过读取 `finance/基金明细跟踪.md` 或其他 Markdown 文件提供业务查询。
- 不要求 agent 为该链路生成 Markdown 文档。
- 不新增投资建议、交易建议、风险评分或自动调仓能力。
- 不处理完整用户登录、角色、家庭账户、共享账户等权限体系；用户权限只影响账户资产查询，不影响后台基金公开数据刷新任务。

### 产品影响范围

- 业务模块：估值与收益、持仓汇总、AI/Agent 数据处理。
- 涉及端：server API、agent 服务接口、前端或外部调用方。
- 用户入口：账户资产详情页或接口调用方；基金明细刷新本身无用户入口，由后台自动执行。
- 相关业务流程：后台调度刷新 -> 任务创建 -> agent 刷新 -> 回调入库 -> 查询资产与基金明细。
- 相关业务对象：处理任务、基金代码、基金详情快照、基金详情项、基金重仓、处理 warning、账户资产。

## 4. 用户角色与主要流程

### 4.1 用户角色

- 普通用户：查看自己的账户资产和已关联的基金公开明细。
- 系统任务：根据全局基金代码集合或调度策略触发基金明细刷新。
- 运维/开发人员：排查刷新失败、回调失败、外部接口波动或数据缺失。

### 4.2 用户故事

- 作为普通用户，我希望在查看账户资产时看到对应基金的最新公开明细，以便理解持仓基金的申购状态、阶段收益和公开重仓。
- 作为系统维护者，我希望基金公开数据由后台自动刷新，以便用户查询资产时尽量看到最新公开明细。
- 作为系统维护者，我希望每次刷新都有任务记录和 warning，以便排查哪些基金刷新失败或数据缺失。

### 4.3 主流程

1. server 后台调度或系统任务发起基金明细刷新。
2. server 根据全局基金代码集合、资产主数据或系统配置中的基金代码创建刷新任务。
3. server 将任务标识、基金代码、回调地址和必要授权信息发送给 agent。
4. agent 刷新基金公开数据。
5. agent 调用 server 回调接口，提交结构化刷新结果。
6. server 校验回调来源、任务状态、幂等标识和契约版本。
7. server 保存基金详情快照、明细、重仓和 warning，并更新任务状态。
8. 用户或调用方通过 server 查询账户资产及对应的全局最新基金明细。

### 4.4 异常流程

- agent 接收任务失败：server 将任务标记为下发失败，并返回可重试状态。
- agent 部分基金刷新失败：server 保留成功基金明细，并将任务标记为部分失败。
- agent 回调失败或重复回调：agent 最多尝试 3 次回调，仍失败则记录 `callback_failed`；server 应支持幂等处理，并保留可排查状态。
- 回调契约版本不支持：server 拒绝落库并记录任务失败原因。
- 查询时某个持仓没有基金明细：server 仍返回账户资产，并明确该基金暂无最新明细。

### 4.5 状态流转

```text
created -> dispatched -> running -> callback_received -> succeeded
created -> dispatched -> running -> callback_received -> partial_failed
created -> dispatch_failed
created -> dispatched -> running -> failed
```

## 5. 需求明细

### 5.1 创建基金刷新任务

- server 应支持后台任务通过显式基金代码列表创建刷新任务。
- server 后续可支持从全局资产主数据、全局已持有基金代码集合或系统配置中提取基金代码创建刷新任务。
- server 任务应记录任务类型、任务状态、来源类型、来源引用、基金代码数量、创建时间和最近更新时间；基金公开数据刷新任务不绑定具体用户。
- server 不应把真实账户名称、资产金额或完整持仓组合发送给 agent，除非后续需求明确要求。

### 5.2 下发 agent 刷新任务

- server 应向 agent 发送任务标识、基金代码列表、契约版本、回调地址和必要授权标识。
- server 应能区分任务下发失败、agent 处理失败和回调处理失败。
- server 应保留 agent 返回或调用失败时的可诊断摘要。

### 5.3 接收 agent 回调

- server 应提供 agent 回调入口，接收结构化基金刷新结果。
- server 应校验回调鉴权、任务标识、幂等键和契约版本。
- server 应支持成功、部分失败和失败三类回调结果。
- server 应保存成功基金的数据，并保存失败基金或缺失字段对应 warning。

### 5.4 查询账户资产与基金明细

- server 应以账户资产和持仓为主事实源返回用户资产。
- server 查询时应先读取当前用户自己的 `portfolio` 持仓，再从持仓中的基金代码或资产代码提取需要关联的 `fund_code`。
- server 应使用这些 `fund_code` 到基金公开数据表中查询全局最新基金详情，并只把匹配当前用户持仓的基金明细拼接到资产结果中。
- 当基金明细不存在、过期或刷新失败时，server 应返回清晰状态，不影响账户资产主体展示。

### 5.5 业务规则

- 基金公开数据只能作为基金详情参考信息，不得反向覆盖用户持仓事实。
- 基金公开数据是全局公开数据，不按用户维度保存；用户查询时只用自己的持仓基金代码关联全局最新明细。
- agent 回调结果是否保存、覆盖、展示、归档或丢弃，由 server 决定。
- 同一任务的重复回调不得重复写入相同快照或造成任务状态回退。
- 不支持的 schema version 必须拒绝处理。

### 5.6 业务数据约束

- 基金代码不能为空，且同一任务内应去重。
- 任务结果必须包含 server 任务标识。
- warning 必须保留机器可识别 code、可读 message 和 severity。
- 基金详情缺失字段应保留缺失原因，不使用空字符串表达未知状态。

### 5.7 权限与可见性

- 可操作角色：系统任务或具备后台管理权限的调用方。
- 可查看角色：资产所属用户或具备后续授权规则的调用方。
- 不可操作或不可查看的场景：用户查看其他用户资产、未授权 agent 回调、任务标识不匹配、回调签名无效。

## 6. 非功能要求

- 性能体验：刷新任务后台异步执行，用户查询不应因外部行情接口耗时而长时间阻塞。
- 安全与合规：server 到 agent 只发送必要基金代码；回调接口必须有鉴权和幂等保护；日志不得打印真实资产明细。
- 可用性：单只基金刷新失败不应导致整个任务完全不可用；应支持部分成功。
- 适用端与设备要求：主要面向 server API 调用方，前端可基于任务状态和查询接口展示。

## 7. 验收标准

- [x] 给定后台任务提供有效基金代码列表，当调度触发刷新时，server 创建任务并向 agent 下发刷新请求。
- [x] 给定 agent 返回成功回调，当 server 校验通过时，server 保存基金详情快照、明细、重仓和 warning，并将任务标记为成功。
- [x] 给定 agent 返回部分失败回调，当部分基金有数据时，server 保存成功部分并将任务标记为部分失败。
- [x] 给定重复回调，当任务和幂等键一致时，server 不重复写入同一结果。
- [x] 给定未授权或契约版本不支持的回调，当 agent 请求 server 时，server 拒绝处理并记录失败原因。
- [x] 给定用户查询账户资产，当用户持仓中的部分基金代码有全局最新明细时，server 返回账户资产和对应基金明细；缺失基金明细不影响资产主体返回。
- [x] 给定 server 查询链路，不读取 agent 本地 Markdown 报告文件。

## 8. 依赖与风险

- 依赖：agent 需要提供 HTTP 任务接口和回调能力。
- 依赖：server 和 agent 需要共享任务请求与结果回调契约。
- 风险：外部公开行情接口不稳定时，任务可能频繁部分失败。
- 风险：异步回调引入重试、幂等、超时和状态一致性复杂度。
- 风险：如果回调鉴权不足，可能造成伪造基金数据写入。

## 9. 未决问题

- [x] agent 回调鉴权暂不做复杂方案，第一阶段采用内部调用约定，后续再健全。
- [x] server 下发后台任务默认允许联网，不记录用户显式联网授权。
- [x] 基金明细“最新”的查询口径：按基金代码取全局最新成功或部分成功中可用的基金明细，不按用户任务维度取数。
- [x] 回调失败最多由 agent 尝试 3 次：立即、10 秒、60 秒；仍失败则记录异常并标记 `callback_failed`，server 不主动轮询补结果，等待下一次后台调度再刷新。

## 10. OpenSpec 衔接

- 是否需要创建 OpenSpec change：是
- 建议 change 名称：`agent-async-fund-refresh`
- 需要进入 spec 的业务规则：任务创建、agent 下发、回调校验、幂等处理、基金详情保存、账户资产与基金明细查询。
- 需要进入 spec 的验收标准：任务状态流转、成功/部分失败/失败处理、未授权回调拒绝、Markdown 不参与查询链路。
- 需要研发在 proposal / design 中评估的问题：数据表、事务边界、回调鉴权、超时重试、跨领域编排、agent 契约版本兼容。
