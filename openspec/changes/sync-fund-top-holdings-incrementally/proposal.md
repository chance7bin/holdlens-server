## Why

当前基金刷新会先删除某基金的全部 `fund_top_holding`，再重新插入最新重仓，导致未变化排名的记录 ID 也被重建，自增序列随每次刷新持续增长。用户希望在保持“只保存当前重仓”语义的前提下，仅对真实新增排名分配新 ID，并删除已经失效的排名。

本变更来源于现有入库逻辑优化，不直接覆盖 `docs/requirements/**/prd-*.md` 中的产品需求，因此无需读取或更新 PRD。

## What Changes

- 将基金重仓保存从“按基金全删后重插”调整为按 `fund_code + rank_no` 差量同步。
- 对仍然存在的排名更新股票及持仓字段，保留原记录 ID。
- 对新增排名插入新记录，仅在真实新增时消耗自增 ID。
- 删除本次结果中已不存在的旧排名，不保留空槽或无效重仓记录。
- 空重仓结果仍表示清空该基金的当前重仓。
- 保持现有回调契约、表结构、唯一键、事务边界和查询返回结构不变。
- 非目标：不让 ID 跟随股票跨排名移动，不保存重仓历史，不承诺自增序列连续或永不增长。
- 成功标准：未变化排名刷新后 ID 不变；新增排名获得新 ID；减少排名后旧记录不存在；同步失败时整次回调事务回滚。

## Capabilities

### New Capabilities

- `fund-top-holding-incremental-sync`: 定义基金当前重仓按基金代码和排名差量更新、插入与清理的持久化行为。

### Modified Capabilities

无。

## Impact

- 影响 `FundDataRepository` 的当前重仓保存编排。
- 影响 `IFundTopHoldingDao` 及其 MyBatis Mapper XML，需要提供查询现有排名、更新已有记录和删除失效排名的能力。
- 需要调整基础设施层单元测试，覆盖同排名更新、新排名插入、失效排名删除、空集合清空和异常回滚相关行为。
- 不修改 HTTP API、agent 回调 payload、领域模型、数据库表结构或外部依赖。
