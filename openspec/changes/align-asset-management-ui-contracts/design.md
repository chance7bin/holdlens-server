## Context

系统目录使用 `user_id = null` 表示全局默认目录，用户目录使用所属 `user_id` 隔离。当前目录是否可记录由“是否存在可见子目录”动态判断，但创建用户子目录时又将除 `INVESTMENT_ASSET` 外的系统目录全部视为叶子，造成模型语义不一致。

资产记录使用 `assetKind + assetRef` 表达可选的公共基金或股票关联。`assetRef` 对客户端不透明，记录列表不能要求客户端从中解析基金代码或股票市场。基金和股票详情属于公共市场事实，不应依赖用户是否存在登记记录。

## Goals / Non-Goals

**Goals:**

- 系统目录作为全局默认目录，用户可以在一级系统目录下创建自己的二级目录。
- 一个用户视角下，同一目录不能同时包含直接记录和子目录。
- 记录响应只返回对应资产类型需要的关联字段。
- 统一市场详情入口只依赖公共标的引用，用户登记关系由其他接口独立查询。

**Non-Goals:**

- 不增加第三级目录、分类排序交互或目录移动 UI。
- 不迁移或自动重分配已经直接挂在系统目录下的用户记录。
- 不实现基金风险等级采集、存储或推导。
- 不删除现有 `/api/funds/{fundCode}` 和 `/api/stocks/detail` 兼容接口。

## Decisions

### 1. 系统目录与用户目录只区分所有权，不硬编码父级白名单

系统目录继续由初始化维护、全局可见且不可编辑；用户目录必须保存 `userId`。创建二级目录时只校验父目录可见、启用、处于一级、金额方向一致，并确认该用户在父目录下没有活跃直接记录。系统目录不再因为 `catalogScope=SYSTEM` 自动禁止用户子目录。

目录是否可直接记录继续由“该用户视角下是否存在可见子目录”决定。因此不同用户可以独立选择直接使用系统目录，或先创建自己的二级目录；一旦已有活跃直接记录，必须先处理记录后才能把目录转换为分组。

### 2. 资产记录使用互斥的基金/股票关联信息

`AssetDTO.Record` 保留外层 `assetKind` 和 `assetRef`，用于区分普通记录、未关联基金/股票记录和已关联记录；新增互斥字段：

- `fund`：仅关联基金时返回 `assetCode`。
- `stock`：仅关联股票时返回 `assetCode`、`assetMarket`、`assetMarketLabel`。

未关联记录的两个字段均为空。响应不增加没有可靠来源或 UI 不需要的股票 `assetType`。

Repository 的查询映射负责从 `fund`、`stock_market` 读取公共标识，Case/Domain 只承载查询结果；Controller 负责市场展示标签映射。创建响应由 Case 在校验公共标的时同步附加同一标识，避免创建后与重新查询结果不一致。

### 3. 统一市场详情由 Case 层编排

新增 `GET /api/market-assets/detail?userId=&assetKind=&assetRef=`。Trigger 只绑定参数和转换 DTO；新的 marketasset Case 校验引用并按资产类型编排既有基金详情和股票详情用例：

- `fund`：根据 `assetRef` 解析基金代码，读取公共基金详情。
- `stock`：根据 `assetRef` 读取公共股票详情。

响应只填充 `fund` 或 `stock` 之一，并在顶层返回 `watchlisted`。该查询不读取 `asset_record`，页面上的“已登记记录”继续通过资产记录接口独立获取。

### 4. 风险等级首期不展示

当前基金数据、agent 回调和数据库均没有可信风险等级字段。客户端原型删除风险等级行，服务端不得由 `fundType` 推导风险等级。

## Migration / Rollback

- 初始化 SQL 将 `BANK_CARD` 名称改为“银行卡”。
- 增加幂等迁移，仅按稳定 `catalog_code` 更新系统目录名称。
- 新详情接口和记录附加字段均为向后兼容新增；回滚时客户端可继续使用旧详情接口。
- 目录规则回滚前需确认没有用户目录已挂在过去不允许的系统目录下。

## Open Questions

当前无待确认事项。系统目录全局共享、用户目录按 `userId` 隔离，基金风险等级首期移除。
