## 背景

当前基金详情、A 股全量行情和美股全量行情的 agent 回调处理都在同一事务中完成：保存回调幂等记录、写入业务结果、更新任务终态。如果业务结果落库失败，例如本地数据库还未执行最新 SQL，事务会回滚，任务仍停留在 `running`，后续重新创建同类型任务会被“任务正在运行”拦截。

这会把一次 server 侧回调处理失败放大成长期占坑问题，运维或开发人员只能手动改库解锁。

## 范围

- 覆盖 `fund_detail_refresh`、`a_share_market_refresh`、`us_stock_market_refresh` 三类已存在回调。
- 当已识别任务和幂等键的回调在 server 处理阶段失败时，将任务可靠标记为 `callback_failed`。
- 记录安全错误摘要，并尽量保留回调处理失败记录。
- 保持重复回调幂等：任务进入终态后，重复回调不再重复写入业务结果。
- 保持创建任务时同类型非终态任务拦截规则不变。

## 非目标

- 不新增任务重试接口。
- 不改造 agent 回调契约、回调鉴权或重试策略。
- 不引入通用任务执行框架。
- 不改变业务数据表结构。
- 不把所有非法回调都改成 `callback_failed`；无法识别任务或缺少幂等键的回调仍按现有校验拒绝。

## 用户可见影响

- 如果 server 已收到某个有效任务回调，但业务落库阶段失败，该任务最终状态会变为 `callback_failed`，不会继续阻塞新的同类型任务创建。
- 重新下发美股全量行情刷新时，不会再因为上一轮 server 回调落库异常而长期报“美股全量行情刷新任务正在运行”。

## 成功标准

- 能复现“美股回调落库异常后任务仍为 running”的测试先失败，再通过。
- 三类回调处理失败后任务均可落到 `callback_failed`。
- `openspec validate --strict mark-callback-processing-failures-terminal` 通过。
- 相关后端测试通过。
